# The map that maintains itself: round two of OKF in my harness

Two weeks ago I wrote about testing the [Open Knowledge Format](/blog/okf-mapa-antes-da-pergunta) in Keel, my harness scaffolder. The idea was simple: instead of the agent starting every session blind, a hook walks the documentation and injects an index before the first turn. The agent already knows what exists and where it is, without burning a tool call to ask.

That was v0.1, and I was honest about what was missing. It was a map you could only read. The agent saw the index, but had no way to write back into the format. It didn't have the reserved files the spec asks for. The links were something I invented that only my hook understood. I promised to post the evolution as it improved. It improved a lot, so here it is.

## From reading the map to drawing it

The biggest hole in v0.1 was this: the agent read, but it didn't contribute. You hand it a new document, it summarizes on the spot, answers, and throws it away. Next session, it forgets everything. It's a librarian who reads every book that arrives and catalogs none of them.

The answer became a new skill, `okf-enrich`. The mental model is what Karpathy called an "LLM wiki": the human curates the sources, decides what deserves to enter the library, and the agent does the librarian's work, distilling that into concept pages, cataloging and cross-referencing.

In practice, you hand it a source (a file, a URL, pasted text, a repo path) and the loop runs:

1. **Read and extract concepts.** Not one page per paragraph, but the entities, decisions, APIs and patterns that deserve to stand on their own.
2. **Survey the types that already exist.** Before inventing a new concept type, reuse one already in the library. It's the same whitelist discipline I apply across the rest of the harness: the allowed types are the ones that already exist, a new type only for something genuinely new. This keeps the library from turning into a mess of synonyms, with `table`, `entity`, `model` and `schema` all for the same thing.
3. **Write or update the pages.** Each concept gets frontmatter (`type`, `title`, `description`, `timestamp`) and a body that **always ends in `# Citations`**, so every claim is traceable to its origin. If the page already exists, it **merges** the new facts in, never clobbers.
4. **Refresh the index and log the change** in `log.md`, the reserved change history the spec asks for, always newest-first.

One rule I insist on: the skill **never commits**. It distills and organizes; the human reviews and decides what stays. Curation stays mine; the grunt work is its.

## An index that lies is worse than no index

v0.1 didn't have the `index.md` the spec reserves for listing a directory. Now it does, generated by a command (`okf-build-index build`) from the concepts that actually exist.

Generating the file is the easy part. The real problem is it rotting. A stale index is that "mall directory" board that still points to a shop that moved floors six months ago: worse than no board at all, because you trust it and walk to the wrong place. If the agent trusts an index that lies, it acts wrong with total confidence.

That's why the generator has a twin: `okf-build-index check`. It regenerates the index in memory and compares it byte for byte against what's on disk. If they diverged, the commit is blocked. The index can't go stale because the commit door won't open while it's stale. The guarantee is mechanical, it doesn't depend on me remembering to run it again.

## One floor at a time

v0.1 had a blunt cap of 60 concepts, existing only so the injection wouldn't blow up the token cost. On a large project, that cut good information before it should.

The way out was what OKF calls *progressive disclosure*, and the analogy is literal. A big building doesn't put every room on a single list in the lobby. The lobby lists the floors; each floor lists its rooms. You go down the hierarchy only as far as you need.

`index.md` now works the same way: instead of one giant list at the root, every directory gets its own. The root carries the OKF version, its own concepts, and a subdirectories section pointing to the child indexes. Each child lists its own and points further down. A large bundle is navigated directory by directory, not as a wall of text. And `check` compares the whole tree, so no floor goes stale.

## No longer speaking only my language

This was the most important conceptual fix, and the most annoying to admit. In v0.1 the links between documents were `[[wiki-links]]`, a syntax I invented that only my hook knew how to read. It was internal jargon. Pretty on paper, useless to any other tool: GitHub doesn't render it, the editor won't click it, the OKF spec itself prefers something else.

I swapped it all for real **relative markdown links**: `[Gate vs skill](gates-vs-skills.md)`. The difference is going from private slang to a common language. Now the same link that forms an edge in the internal graph is also clickable on GitHub, resolves in any markdown renderer, and survives outside Keel. I no longer need a translator to read my own documents.

The side effect was honest and measurable: when I cross-linked Keel's own docs with real links, the graph went from zero edges to seven. It wasn't a tool failure, since link detection was already there, covered by tests. It was a content failure: I had documents that were related and never said so to each other.

## And you can see the graph

I closed with a visualizer, `okf-visualize`. It reads the concepts and their cross-links and spits out **one** self-contained HTML file: a force-directed graph in plain JavaScript, nodes colored by type, with a legend and hover descriptions. Zero external URLs, opening straight from `file://`, no server.

The official OKF visualizer depends on GCP and Python. Mine runs offline, on any bundle, with nothing to install. It's the subway map of the project's knowledge: one glance and you see the clusters, the isolated nodes, what should connect and doesn't.

## Where this version still falls short

I stay honest about the limits. The frontmatter parser is still naive: `key: value` lines, no nested structure. The type-whitelist discipline depends on the agent respecting existing types; the lint catches orphans and contradictions, but the one who really curates is me. And the graph is still thin: seven edges beats zero, but it's a project I documented bit by bit, and a lot of the knowledge still lives in my head, not in a cross-linked page. The tool matured faster than the content it organizes.

None of that bothers me. The cycle is this: I ship the piece, use it, the friction shows up, I adjust. v0.1 was a map you could read. This one is a map you can write, that maintains itself, and that you can see. The next one tightens what's still missing.

## What I take from this

The usual pattern: context the agent already has by default beats context that only shows up if it remembers to ask. What's different this round is that the map now maintains itself. The agent helps draw it, a gate stops it from lying, and the links speak a language the whole world understands. The spec is still new, my implementation moves fast. If you're also working with OKF, or hit the same limits, I want to hear about it.
